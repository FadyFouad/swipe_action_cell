# Implementation Plan: Right-Swipe Progressive Action

**Branch**: `003-progressive-swipe` | **Date**: 2026-02-25 | **Spec**: [spec.md](spec.md)

## Summary

Add right-swipe progressive (incremental) action behavior to `SwipeActionCell`. When the
user swipes right past the activation threshold and releases, a cumulative numeric value
is incremented by the configured step and all change callbacks fire. The cell snaps back
to idle immediately — it never enters `SwipeState.revealed` in progressive mode. New
public types: `OverflowBehavior`, `ProgressIndicatorConfig`, `ProgressiveSwipeConfig`,
`ProgressiveSwipeIndicator`. Updated: `SwipeActionCell` (new `rightSwipe` parameter),
barrel exports.

## Implementation Notes

### Architecture Decisions

**State machine hook — right progressive vs. left reveal**

The existing `_handleAnimationStatusChange`:
```dart
} else if (_state == SwipeState.animatingToOpen) {
  _updateState(SwipeState.revealed);  // existing path (left swipe / no progressive config)
}
```
Becomes:
```dart
} else if (_state == SwipeState.animatingToOpen) {
  if (_lockedDirection == SwipeDirection.right && widget.rightSwipe != null) {
    _applyProgressiveIncrement();       // new: increment value, fire callbacks
    _isPostIncrementSnapBack = true;
    _updateState(SwipeState.animatingToClose);
    _snapBack(_controller.value, 0.0);  // snap back to idle
  } else {
    _updateState(SwipeState.revealed);  // unchanged: left swipe reveal
  }
}
```

The `animatingToClose → idle` transition then checks `_isPostIncrementSnapBack` to
avoid firing `onSwipeCancelled` for the post-increment snap-back.

**ValueNotifier lifecycle**

```dart
@override
void initState() {
  super.initState();
  // existing: _controller setup ...
  if (widget.rightSwipe != null) {
    final config = widget.rightSwipe!;
    final controlled = config.value != null;
    final initialVal = controlled
        ? config.value!
        : config.initialValue.clamp(config.minValue, config.maxValue);
    _progressValueNotifier = ValueNotifier(initialVal);
  }
}

@override
void didUpdateWidget(SwipeActionCell oldWidget) {
  super.didUpdateWidget(oldWidget);
  final newConfig = widget.rightSwipe;
  final oldConfig = oldWidget.rightSwipe;
  if (newConfig != null) {
    _progressValueNotifier ??= ValueNotifier(
      newConfig.initialValue.clamp(newConfig.minValue, newConfig.maxValue),
    );
    // Mirror external value in controlled mode
    if (newConfig.value != null && newConfig.value != _progressValueNotifier!.value) {
      _progressValueNotifier!.value = newConfig.value!;
    }
  } else if (oldConfig != null && newConfig == null) {
    _progressValueNotifier?.dispose();
    _progressValueNotifier = null;
  }
}

@override
void dispose() {
  _progressValueNotifier?.dispose();
  // existing: _controller dispose ...
  super.dispose();
}
```

**`_computeNextValue` — pure function (co-located in widget file or separate)**

```dart
/// Computes the next cumulative value after a successful right swipe.
///
/// Returns a record of the constrained next value and a bool indicating
/// whether [maxValue] was reached or exceeded.
({double nextValue, bool hitMax}) _computeNextValue({
  required double current,
  required ProgressiveSwipeConfig config,
}) {
  final step = config.dynamicStep != null
      ? config.dynamicStep!(current)
      : config.stepValue;

  if (step <= 0) return (nextValue: current, hitMax: false);

  final candidate = current + step;

  return switch (config.overflowBehavior) {
    OverflowBehavior.clamp => (
        nextValue: candidate.clamp(config.minValue, config.maxValue),
        hitMax: candidate >= config.maxValue,
      ),
    OverflowBehavior.wrap => candidate > config.maxValue
        ? (nextValue: config.minValue, hitMax: true)
        : (nextValue: candidate, hitMax: false),
    OverflowBehavior.ignore => (nextValue: candidate, hitMax: false),
  };
}
```

Uses Dart 3 records and exhaustive switch — requires Dart >=3.4.0 (Constitution).

**`_applyProgressiveIncrement` — integrated into state**

```dart
void _applyProgressiveIncrement() {
  final config = widget.rightSwipe!;
  final current = _progressValueNotifier!.value;
  final (:nextValue, :hitMax) = _computeNextValue(current: current, config: config);

  final isControlled = config.value != null;

  if (nextValue != current) {
    if (!isControlled) {
      _progressValueNotifier!.value = nextValue;
    }
    config.onProgressChanged?.call(nextValue, current);
  }
  if (hitMax) config.onMaxReached?.call();
  if (config.enableHaptic) HapticFeedback.mediumImpact();
  config.onSwipeCompleted?.call(nextValue);
}
```

**Progress indicator in build()**

```dart
// Inside _wrapWithProgressIndicator(Widget child):
Widget _buildProgressIndicator() {
  final config = widget.rightSwipe!;
  final indicatorConfig = config.progressIndicatorConfig
      ?? const ProgressIndicatorConfig();

  assert(
    !config.maxValue.isInfinite,
    'showProgressIndicator requires a finite maxValue',
  );

  if (config.maxValue.isInfinite) return const SizedBox.shrink();

  return Positioned(
    top: 0,
    bottom: 0,
    left: 0,
    width: indicatorConfig.width,
    child: RepaintBoundary(
      child: ValueListenableBuilder<double>(
        valueListenable: _progressValueNotifier!,
        builder: (context, value, _) {
          final fillRatio = (value / config.maxValue).clamp(0.0, 1.0);
          return ProgressiveSwipeIndicator(
            fillRatio: fillRatio,
            config: indicatorConfig,
          );
        },
      ),
    ),
  );
}
```

### Performance Considerations

- `ValueListenableBuilder` isolates indicator rebuilds: only the `ProgressiveSwipeIndicator`
  subtree repaints on value change. The gesture detector and child do not rebuild.
- `RepaintBoundary` wraps the indicator so its `CustomPaint` repaints in its own layer.
- `CustomPainter.shouldRepaint` returns `true` only when `fillRatio` changes.
- `_computeNextValue` is pure with no allocations beyond the record return value.
- Haptic calls are fire-and-forget — they invoke a platform channel method and return
  immediately without blocking the animation loop.
- `_isPostIncrementSnapBack`, `_hapticThresholdFired`, `_swipeStartedFired` are all
  simple `bool` fields — zero allocation overhead.

### Test Strategy

Tests are written RED first (failing), then implementation makes them GREEN.

**Unit tests** (`test/actions/progressive/progressive_value_logic_test.dart`):
- `_computeNextValue` with clamp/wrap/ignore across boundary values
- Fixed step vs. dynamic step precedence
- Dynamic step returning 0 and negative values (no-op)
- `initialValue` clamping on initialization
- Config equality and `copyWith`

**Widget tests** (`test/widget/swipe_action_cell_progressive_test.dart`):
- Single swipe past threshold → value increments by `stepValue`
- Multiple sequential swipes → values accumulate correctly
- Swipe below threshold → value unchanged, `onSwipeCancelled` fires
- Fling → increment fires (velocity path, not distance path)
- `overflowBehavior: clamp` → value stops at `maxValue`, `onMaxReached` fires
- `overflowBehavior: wrap` → value resets to `minValue`, `onMaxReached` fires
- `overflowBehavior: ignore` → value exceeds `maxValue`, `onMaxReached` not fired
- `dynamicStep` → step varies correctly per swipe
- Callbacks fire in correct order: `onSwipeStarted` → `onProgressChanged` → `onMaxReached` → `onSwipeCompleted`
- Controlled mode → widget does not self-update; `onProgressChanged` fires with correct values
- Uncontrolled mode → widget updates internally; indicator reflects new value
- Progress indicator renders at correct fill ratio
- Progress indicator updates after swipe completion
- `rightSwipe: null` → no progressive behavior, cell reverts to F001/F002 behavior
- Rapid swipes during animation → ignored (FR-019), no double-increment
- Haptic flags correct (mocked via `TestDefaultBinaryMessengerBinding`)
