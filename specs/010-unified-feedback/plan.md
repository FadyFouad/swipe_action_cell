# Implementation Plan: Unified Feedback System

**Branch**: `010-unified-feedback` | **Date**: 2026-03-01 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/010-unified-feedback/spec.md`

---

## Summary

Consolidate all scattered `HapticFeedback.*` calls in the `swipe_action_cell` package into a single `FeedbackDispatcher` service driven by an immutable `SwipeFeedbackConfig`. Introduce per-event haptic pattern overrides via `HapticPattern` (1–8 `HapticStep` sequences), a multi-step timer management system, an audio hook callback (`onShouldPlaySound`), and theme-level propagation via `SwipeActionCellTheme.feedbackConfig`. All legacy `enableHaptic` boolean flags on direction configs are preserved for backward compatibility; the new config takes precedence when present.

## Key Implementation Notes

### `_hapticThresholdFired` Consolidation

The two separate direction-guarded checks (lines 1115–1130) collapse into one direction-agnostic check. The `isForward` flag is passed to the dispatcher so legacy mode can apply the correct direction flag:

```dart
if (progress.isActivated && !_hapticThresholdFired && !_hasActiveZones()) {
  _feedbackDispatcher?.fire(
    SwipeFeedbackEvent.thresholdCrossed,
    isForward: _dragIsForward,
  );
  _hapticThresholdFired = true;
}
```

### `_fireZoneHaptic` Preservation

`_fireZoneHaptic(SwipeZoneHaptic?)` is kept unchanged. It is only bypassed when a `FeedbackDispatcher` is active. F009's `SwipeZone.hapticPattern` field continues to work for consumers who do not adopt `SwipeFeedbackConfig` (FR-020 compliance).

### Platform Safety Pattern

Every `HapticFeedback.*` call inside `FeedbackDispatcher._fireHapticType()`:

```dart
void _fireHapticType(HapticType type) {
  try {
    switch (type) {
      case HapticType.lightImpact:       HapticFeedback.lightImpact();
      case HapticType.mediumImpact:      HapticFeedback.mediumImpact();
      case HapticType.heavyImpact:       HapticFeedback.heavyImpact();
      case HapticType.successNotification: HapticFeedback.vibrate();
      case HapticType.errorNotification:   HapticFeedback.vibrate();
      case HapticType.selectionTick:     HapticFeedback.selectionClick();
    }
  } catch (Object _) {
    // Platform does not support haptic; skip silently (FR-021)
  }
}
```

Note: `successNotification` and `errorNotification` both map to `HapticFeedback.vibrate()` in this implementation (Flutter SDK does not expose notification-level granularity on all platforms). On iOS, the notification feedback types are mapped via the underlying engine; calling `vibrate()` is the cross-platform safe fallback.

### Panel Feedback Threading

The widget creates the `SwipeActionPanel` with:
```dart
SwipeActionPanel(
  actions: config.actions,
  panelWidth: panelWidth,
  onClose: _closePanel,
  enableHaptic: config.enableHaptic,  // preserved for backward compat
  onFeedbackRequest: _feedbackDispatcher != null
    ? () => _feedbackDispatcher!.fire(SwipeFeedbackEvent.actionTriggered)
    : null,
)
```

When `_feedbackDispatcher` is non-null, `onFeedbackRequest` overrides `enableHaptic` in the panel.

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Existing haptic tests rely on direct `HapticFeedback.*` mocking | Keep same mock channel approach; dispatcher uses the same underlying API |
| `_hapticThresholdFired` logic change breaks existing threshold tests | Combined condition is functionally equivalent; test both direction paths |
| `SwipeActionPanel.enableHaptic` is public — adding parameter must not break callers | `onFeedbackRequest` defaults to null; existing `enableHaptic` path unchanged when null |
| Multi-step timer fires after widget dispose | `cancelPendingTimers()` in `dispose()` prevents post-unmount execution |
| Zone haptic migration: legacy `_fireZoneHaptic` must still work | Preserved; guarded by null check on dispatcher |
